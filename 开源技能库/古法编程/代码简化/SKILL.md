# Code Simplification（代码简化）

> 受 [Claude Code Simplifier 插件](https://github.com/anthropics/claude-plugins-official/blob/main/plugins/code-simplifier/agents/code-simplifier.md) 启发。在此改编为适用于任何 AI 编码 Agent 的模型无关、流程驱动技能。

## 概述

通过减少复杂度来简化代码，同时保持精确行为。目标不是更少的行数——是代码更容易读、理解、修改和调试。每个简化必须通过一个简单测试："新团队成员是否能比原始版本更快地理解这个？"

## 什么时候用

- 功能已经工作且测试通过，但实现比需要的更重
- 代码审查中标记了可读性或复杂度问题
- 遇到深层嵌套逻辑、长函数或不清晰的名称
- 重构在时间压力下写的代码
- 整合分散在文件中的相关逻辑
- 合并引入了重复或不一致

**什么时候不用：**

- 代码已经干净可读——不要为简化而简化
- 你还不理解代码做什么——先理解再简化
- 代码是性能关键的且"更简单"版本会明显更慢
- 你即将完全重写该模块——简化即将丢弃的代码浪费精力

## 五原则

### 1. 精确保留行为

不改代码做什么——只改它怎么表达。所有输入、输出、副作用、错误行为和边界情况必须保持相同。

```
每次变更前问：
→ 对每个输入产生相同输出？
→ 保持相同的错误行为？
→ 保持相同的副作用和顺序？
→ 所有已有测试不修改就通过？

```


### 2. 遵循项目规范

简化意味着让代码更一致于代码库，不是强加外部偏好。

### 3. 清晰优先于巧妙

需要暂停解析的紧凑代码不如显式代码好。

```typescript
// 不清晰：密集三元链
const label = isNew ? 'New' : isUpdated ? 'Updated' : isArchived ? 'Archived' : 'Active';

// 清晰：可读映射
function getStatusLabel(item: Item): string {
 if (item.isNew) return 'New';
 if (item.isUpdated) return 'Updated';
 if (item.isArchived) return 'Archived';
 return 'Active';
}

```


### 4. 保持平衡

简化有失败模式：过度简化。注意这些陷阱：

- **过度内联** — 删除给了概念名称的辅助函数让调用点更难读
- **合并无关逻辑** — 两个简单函数合成一个复杂函数不是更简单
- **删除"不必要"的抽象** — 某些抽象是为了可扩展性或可测试性

### 5. 限定在变更范围

默认只简化最近修改的代码。除非明确要求扩大范围，不要顺手重构无关代码。

## 简化流程

### 步骤 1：理解再动手（Chesterton's Fence）

改变或删除任何东西之前，理解它为什么存在：如果你看到路上有栅栏但不理解为什么在那里，不要拆掉它。先理解原因，然后决定原因是否仍然成立。

### 步骤 2：识别简化机会

扫描这些模式：

**结构复杂度：**

| 模式 | 信号 | 简化 |
|------|------|------|
| 深层嵌套（3+ 层） | 控制流难跟 | 提取条件到 guard 子句或辅助函数 |
| 长函数（50+ 行） | 多个职责 | 拆成聚焦函数，用描述性名称 |
| 嵌套三元 | 需要心智栈解析 | 替换为 if/else、switch 或查找对象 |

**命名和可读性：**

| 模式 | 信号 | 简化 |
|------|------|------|
| 通用名 | `data`、`result`、`temp` | 改名描述内容 |
| 缩写名 | `usr`、`cfg`、`btn` | 用全称除非缩写是通用的 |
| 注释解释"what" | `// increment counter` 在 `count++` 上方 | 删除注释 |
| 注释解释"why" | `// Retry because the API is flaky` | 保留这些 |

**冗余：**

| 模式 | 信号 | 简化 |
|------|------|------|
| 重复逻辑 | 多处相同的 5+ 行 | 提取为共享函数 |
| 死代码 | 不可达分支、未使用变量 | 删除 |
| 不必要的抽象 | 没有价值的包装器 | 内联包装器 |

### 步骤 3：增量应用变更

一次做一个简化。每次变更后跑测试。

**500 规则：** 如果重构会修改超过 500 行，投资自动化（codemod、AST 变换）而不是手动修改。

### 步骤 4：验证结果

```
比较前后：
- 简化版本真的更容易理解吗？
- 引入了与代码库不一致的新模式？
- diff 干净且可审查？

```


## 语言特定指南

### TypeScript / JavaScript

```typescript
// 简化：不必要的 async 包装
async function getUser(id: string): Promise<User> {
 return await userService.findById(id);
}
// 改后
function getUser(id: string): Promise<User> {
 return userService.findById(id);
}

// 简化：冗余布尔返回
function isValid(input: string): boolean {
 if (input.length > 0 && input.length < 100) { return true; }
 return false;
}
// 改后
function isValid(input: string): boolean {
 return input.length > 0 && input.length < 100;
}

```


## 常见合理化

| 合理化说法 | 现实 |
|---|---|
| "它能跑，不用动" | 能跑但难读的代码坏了时更难修。现在简化省未来每次变更的时间。 |
| "更少行总是更简单" | 1 行嵌套三元不比 5 行 if/else 更简单。简洁是关于理解速度，不是行数。 |
| "类型让它自解释" | 类型记录结构，不记录意图。命名良好的函数比类型签名更好解释 *why*。 |

## 危险信号

- 简化需要修改测试才能通过（你可能改了行为）
- "简化"后的代码比原始版更长更难跟
- 改名匹配你的偏好而不是项目规范
- 删除错误处理因为"让代码更干净"
- 简化你不完全理解的代码

## 验证

完成简化后：

- [ ] 所有已有测试不修改就通过
- [ ] 构建成功无新警告
- [ ] 每个简化是可审查的增量变更
- [ ] diff 干净——没有混入无关变更
- [ ] 简化代码遵循项目规范
- [ ] 没有删除或削弱错误处理
- [ ] 没有留下死代码
