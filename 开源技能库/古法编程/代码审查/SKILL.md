# Code Review and Quality（代码审查和质量）

## 概述

多维度代码审查，配质量门禁。每个变更在合并前都必须审查——没有例外。审查覆盖五个维度：正确性、可读性、架构、安全和性能。

**批准标准：** 当一个变更明确改善了整体代码健康度时就批准，即使它不完美。完美代码不存在——目标是持续改进。不要因为它不是你写的那样就阻塞。如果它改善了代码库并遵循项目规范，就批准。

## 什么时候用

- 合并任何 PR 或变更之前
- 完成功能实现后
- 当另一个 Agent 或模型产出的代码你需要评估时
- 重构已有代码时
- 任何 bug 修复后（审查修复本身和回归测试）

## 五维度审查

每次审查都在这些维度上评估代码：

### 1. 正确性

代码做了它声称做的事吗？

- 匹配规格或任务需求吗？
- 处理了边界情况（null、空、边界值）吗？
- 处理了错误路径（不只是 happy path）吗？
- 通过了所有测试？测试确实在测正确的东西吗？
- 有 off-by-one 错误、竞态条件或状态不一致吗？

### 2. 可读性和简洁性

另一个工程师（或 Agent）能否在作者不解释的情况下理解这段代码？

- 名称描述性和一致性？（没有无上下文的 `temp`、`data`、`result`）
- 控制流直白（避免嵌套三元、深层回调）？
- 代码逻辑组织（相关代码分组、清晰模块边界）？
- 有应该简化的"聪明"技巧？
- **能用更少行实现吗？**（1000 行但 100 行就够了 = 失败）
- **抽象对得起复杂度吗？**（不要在第三个用例之前泛化）
- 注释能帮助澄清非明显意图？（但不要注释明显代码）
- 有死代码残留：无操作变量（`_unused`）、向后兼容垫片、或 `// removed` 注释？

### 3. 架构

变更适合系统设计吗？

- 遵循已有模式还是引入新模式？如果是新的，有正当理由吗？
- 维护了清晰的模块边界？
- 有应该共享的代码重复？
- 依赖方向正确（无循环依赖）？
- 抽象级别合适（不过度工程、不太耦合）？

### 4. 安全

详细安全指南参见 `security-and-hardening`。变更引入了漏洞吗？

- 用户输入验证和清理了？
- 密钥不在代码、日志和版本控制中？
- 需要的地方检查了认证/授权？
- SQL 查询参数化（无字符串拼接）？
- 输出编码防止 XSS？
- 依赖来自可信来源且无已知漏洞？
- 外部数据（API、日志、用户内容、配置文件）被视为不受信任的？

### 5. 性能

详细性能分析和优化参见 `performance-optimization`。变更引入了性能问题吗？

- 有 N+1 查询模式？
- 有无界循环或无约束数据获取？
- 有应该是异步的同步操作？
- UI 组件有不必要的重渲染？
- 列表端点缺少分页？
- 热路径创建了大对象？

## 变更大小

小的、聚焦的变更更容易审查、更快合并、更安全部署。目标大小：

```
~100 行变更 → 好。一次坐下可审查。
~300 行变更 → 如果是单一逻辑变更则可接受。
~1000 行变更 → 太大。拆分。

```


## 变更描述

每个变更都需要一个在版本控制历史中独立存在的描述。

**第一行：** 短、祈使句、独立可理解。"Delete the FizzBuzz RPC" 不是 "Deleting the FizzBuzz RPC"。必须信息量足够让搜索历史的人不读 diff 就能理解变更。

**正文：** 改了什么和为什么。包含代码本身看不到的上下文、决策和推理。链接 bug 编号、基准测试结果或设计文档。承认方案的不足。

**反模式：** "Fix bug"、"Fix build"、"Add patch"、"Moving code from A to B"、"Phase 1"、"Add convenience functions"。

## 审查流程

### 步骤 1：理解上下文

看代码之前，先理解意图：

```
- 这个变更试图完成什么？
- 它实现了什么规格或任务？
- 预期的行为变化是什么？

```


### 步骤 2：先审查测试

测试揭示意图和覆盖度：

```
- 有测试吗？
- 测的是行为（不是实现细节）吗？
- 覆盖了边界情况？
- 测试名描述性强？
- 如果代码变了测试能抓住回归？

```


### 步骤 3：审查实现

用五个维度走查代码：

```
每个修改的文件：
1. 正确性：这段代码做了测试说它应该做的吗？
2. 可读性：我能不靠帮助就理解它吗？
3. 架构：它适合系统吗？
4. 安全：有漏洞吗？
5. 5. 性能：有瓶颈吗？

```


### 步骤 4：分类发现

每条评论标注严重性，让作者知道哪些是必需的、哪些是可选的：

| 前缀                          | 含义       | 作者行动                     |
| ----------------------------- | ---------- | ---------------------------- |
| _(无前缀)_                    | 必需变更   | 合并前必须处理               |
| **Critical:**                 | 阻塞合并   | 安全漏洞、数据丢失、功能损坏 |
| **Nit:**                      | 小的、可选 | 作者可以忽略——格式、风格偏好 |
| **Optional:** / **Consider:** | 建议       | 值得考虑但非必需             |
| **FYI**                       | 仅供参考   | 不需要行动                   |

### 步骤 5：验证验证过程

检查作者的验证流程：

```
- 运行了什么测试？
- 构建通过了？
- 手动测试了吗？
- UI 变更有截图？
- 有前后对比？

```


## 多模型审查模式

用不同模型做不同审查视角：

```
Model A 写代码
 │
 ▼
Model B 审查正确性和架构
 │
 ▼
Model A 处理反馈
 │
 ▼
人类做最终决定

```


## 死代码卫生

任何重构或实现变更后，检查孤立代码：

1. 确定现在不可达或未使用的代码
2. 显式列出
3. **删除前先问：** "应该删除这些现在未使用的元素吗：[list]？"

## 处理分歧

解决审查争议时，应用这个层级：

1. **技术事实和数据**优先于观点和偏好
2. **风格指南**在风格问题上是绝对权威
3. **软件设计**必须按工程原则评估，不按个人偏好
4. **代码库一致性**如果不过度降低健康度是可以接受的

**不要接受"我以后再清理。"** 经验表明延迟清理很少发生。要求在提交前清理，除非是真正的紧急情况。

## 审查中的诚实

审查代码时——不管是自己、另一个 Agent 还是人类写的：

- **不要橡皮图章。** 没有证据的"LGTM"对谁都没帮助。
- **不要淡化真正的问题。** "这可能是小问题"但它是一个会打到生产的 bug 是不诚实的。
- **尽可能量化问题。** "这个 N+1 查询会为列表中每项增加约 50ms"比"这可能会慢"好。
- **对有明确问题的方案说不。** 审查中的讨好是一种失败模式。如果实现有问题，直接说出来并提出替代方案。
- **优雅地接受被推翻。** 如果作者有完整上下文并不同意，尊重他们的判断。评论代码，不评论人。

## 常见合理化

| 合理化说法                | 现实                                                         |
| ------------------------- | ------------------------------------------------------------ |
| "它能跑，够了"            | 能跑但不可读、不安全或架构错误的代码会产生复利增长的技术债。 |
| "我写的所以我知道它对"    | 作者对自己的假设是盲的。每个变更都受益于另一双眼睛。         |
| "我们以后再清理"          | "以后"永远不会来。审查是质量门禁——用它。                     |
| "AI 生成的代码应该没问题" | AI 代码需要更多审查，不是更少。它自信且看似合理，即使错了。  |
| "测试过了所以没问题"      | 测试必要但不充分。它们抓不到架构问题、安全问题或可读性问题。 |

## 危险信号

- PR 没有任何审查就被合并
- 只检查测试是否通过的审查（忽略其他维度）
- 没有实际审查证据的"LGTM"
- 安全敏感变更没有安全聚焦审查
- "太大审不过"的 PR（拆分它们）
- Bug 修复 PR 没有回归测试
- 审查评论没有严重性标签
- 接受"以后再修"——它永远不会发生

## 验证

审查完成后：

- [ ] 所有 Critical 问题已解决
- [ ] 所有 Important 问题已解决或明确延期并附理由
- [ ] 测试通过
- [ ] 构建成功
- [ ] 验证流程已记录（改了什么、怎么验证的）
