# Deprecation and Migration（废弃和迁移）

## 概述

代码是负债，不是资产。每行代码都有持续维护成本——修复 bug、更新依赖、安全补丁、新工程师上手。废弃是移除不再值回投入的代码的纪律，迁移是安全地把用户从旧系统搬到新系统的过程。

大多数工程团队擅长构建东西。很少擅长移除。这个技能解决这个缺口。

## 什么时候用

- 用新系统替换旧系统、API 或库
- 下线不再需要的功能
- 整合重复实现
- 移除没人拥有但所有人都依赖的死代码
- 规划新系统的生命周期（废弃规划从设计时开始）
- 决定维护遗留系统还是投资迁移

## 核心原则

### 代码是负债

每行代码有持续成本：测试、文档、安全补丁、依赖更新和任何人处理它时的心智开销。代码的价值是它提供的功能，不是代码本身。当同样的功能可以用更少代码、更少复杂度或更好的抽象提供——旧代码应该消失。

### Hyrum's Law 让移除变难

用户足够多时，每个可观察行为都会被人依赖——包括 bug、时序怪癖和未文档化的副作用。这就是为什么废弃需要主动迁移，而不仅仅是公告。

### 废弃规划从设计时开始

构建新东西时问："我们 3 年后怎么移除它？"设计有干净接口、feature flag 和最小表面区域的系统比到处泄漏实现细节的系统更容易废弃。

## 废弃决策

废弃任何东西之前，回答这些问题：

```
1. 这个系统仍然提供独特价值？
 → 如果是，维护它。如果否，继续。

2. 有多少用户/消费者依赖它？
 → 量化迁移范围。

3. 有替代品吗？
 → 如果没有，先建替代品。不要没有替代方案就废弃。

4. 每个消费者的迁移成本是多少？
 → 如果可自动化，做。如果手动且高成本，对比维护成本权衡。

5. 不废弃的持续维护成本是什么？
 → 安全风险、工程师时间、复杂度的机会成本。

```


## 强制 vs 建议废弃

| 类型 | 何时用 | 机制 |
|------|-------|------|
| **建议** | 迁移可选，旧系统稳定 | 警告、文档、提示。用户按自己时间线迁移。 |
| **强制** | 旧系统有安全问题、阻碍进步、或维护成本不可持续 | 硬截止日期。提供迁移工具。 |

**默认用建议。** 只在维护成本或风险足以证明强制迁移时使用强制。强制废弃需要提供迁移工具、文档和支持——你不能只宣布一个截止日期。

## 迁移流程

### 步骤 1：构建替代品

没有可工作的替代品就不要废弃。替代品必须：

- 覆盖旧系统的所有关键用例
- 有文档和迁移指南
- 在生产中验证过

### 步骤 2：公告和文档

```markdown
## Deprecation Notice: OldService

**Status:** Deprecated as of 2025-03-01
**Replacement:** NewService (see migration guide below)
**Removal date:** Advisory — no hard deadline yet
**Reason:** OldService requires manual scaling and lacks observability.

### Migration Guide
1. Replace `import { client } from 'old-service'` with `import { client } from 'new-service'`
2. Update configuration
3. Run migration verification: `npx migrate-check`

```


### 步骤 3：增量迁移

逐个迁移消费者，不要一次全部：

```
1. 识别所有与废弃系统的触点
2. 更新为使用替代品
3. 验证行为匹配（测试、集成检查）
4. 移除对旧系统的引用
5. 确认无回归

```


**Churn 规则：** 如果你拥有被废弃的基础设施，你有责任迁移用户——或提供不需要迁移的向后兼容更新。

### 步骤 4：移除旧系统

所有消费者迁移完成之后：

```
1. 验证零活跃使用（指标、日志、依赖分析）
2. 移除代码
3. 移除相关测试、文档和配置
4. 移除废弃公告
5. 庆祝——移除代码是成就

```


## 迁移模式

### 绞杀者模式

新旧系统并行运行。增量地把流量从旧路由到新。旧系统处理 0% 流量时移除它。

### 适配器模式

创建一个适配器把旧接口的调用翻译到新实现。消费者继续用旧接口，你迁移后端。

```typescript
class LegacyTaskService implements OldTaskAPI {
 constructor(private newService: NewTaskService) {}

 getTask(id: number): OldTask {
 const task = this.newService.findById(String(id));
 return this.toOldFormat(task);
 }
}

```


### Feature Flag 迁移

用 feature flag 逐个把消费者从旧系统切换到新系统。

## 僵尸代码

僵尸代码是没人拥有但所有人都依赖的代码。没有活跃维护者，没有明确归属，累积安全漏洞和兼容性问题。

- 6 个月以上无提交但有活跃消费者
- 没有分配维护者或团队
- 没人修的失败测试
- 有已知漏洞的依赖没人更新

**应对：** 要么分配归属并正确维护，要么废弃并有具体迁移计划。僵尸代码不能留在灰色地带——它要么得到投资，要么被移除。

## 常见合理化

| 合理化说法 | 现实 |
|---|---|
| "它还能跑，为什么要移除" | 没人维护的能跑代码会累积安全债和复杂度。 |
| "可能有人以后需要" | 如果以后需要可以重建。保留不用的代码"以防万一"比重建更贵。 |
| "迁移太贵了" | 对比迁移成本和 2-3 年的持续维护成本。迁移通常长期更便宜。 |
| "用户自己会迁移" | 他们不会。提供工具、文档和激励——或自己做迁移。 |

## 危险信号

- 废弃的系统没有替代品
- 废弃公告没有迁移工具或文档
- 多年一直是"建议"的"软"废弃
- 没有归属和活跃消费者的僵尸代码
- 在废弃系统上添加新功能
- 没有测量当前使用就废弃
- 没验证零活跃消费者就移除代码

## 验证

完成废弃后：

- [ ] 替代品已在生产验证，覆盖所有关键用例
- [ ] 存在带具体步骤和示例的迁移指南
- [ ] 所有活跃消费者已迁移（通过指标/日志验证）
- [ ] 旧代码、测试、文档和配置已完全移除
- [ ] 代码库中没有对废弃系统的引用
- [ ] 废弃公告已移除
