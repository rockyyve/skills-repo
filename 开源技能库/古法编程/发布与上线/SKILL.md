# Shipping and Launch（发布和上线）

## 概述

带着信心发布。目标不只是部署——是安全部署，有监控到位、回滚方案就绪、对成功是什么有清晰理解。每次上线都应该是可逆的、可观察的和增量的。

## 什么时候用

- 首次部署功能到生产
- 向用户发布重大变更
- 迁移数据或基础设施
- 开放 beta 或早期访问计划
- 任何有风险的部署（所有）

## 上线前检查清单

### 代码质量

- [ ] 所有测试通过（单元、集成、e2e）
- [ ] 构建成功无警告
- [ ] Lint 和类型检查通过
- [ ] 代码审查并批准
- [ ] 上线前应解决的 TODO 已清
- [ ] 生产代码中没有 `console.log` 调试语句
- [ ] 错误处理覆盖预期失败模式

### 安全

- [ ] 代码和版本控制中没有密钥
- [ ] `npm audit` 无 critical 或 high 漏洞
- [ ] 所有用户面向端点有输入验证
- [ ] 认证和授权检查到位
- [ ] 安全头已配置
- [ ] 认证端点有限流
- [ ] CORS 配置为特定源（非通配符）

### 性能

- [ ] Core Web Vitals 在"好"阈值
- [ ] 关键路径没有 N+1 查询
- [ ] 图片已优化
- [ ] Bundle 大小在预算内
- [ ] 数据库查询有适当索引

### 基础设施

- [ ] 环境变量已在生产设置
- [ ] 数据库迁移已应用
- [ ] DNS 和 SSL 已配置
- [ ] 日志和错误报告已配置
- [ ] 健康检查端点存在并响应

## Feature Flag 策略

在 feature flag 后面发布以解耦部署和发布：

```typescript
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
 return <TaskSharingPanel task={task} />;
}

return null;

```


**Feature flag 生命周期：**

```
1. 部署 flag OFF → 代码在生产但不活跃
2. 团队/beta 启用 → 生产环境中内部测试
3. 逐步上线 → 5% → 25% → 50% → 100%
4. 每阶段监控 → 观察错误率、性能、用户反馈
5. 清理 → 全量后移除 flag 和死代码

```


**规则：**

- 每个 feature flag 有归属和过期日期
- 全量后 2 周内清理 flag
- 不要嵌套 feature flag
- CI 中测试 flag 的两种状态

## 分阶段上线

```
1. 部署到 staging
 └── staging 环境中完整测试
 └── 手动冒烟测试关键流程

2. 部署到生产（flag OFF）
 └── 验证部署成功（健康检查）
 └── 检查错误监控（无新错误）

3. 团队启用（flag ON for 内部用户）
 └── 团队在生产中使用功能
 └── 24 小时监控窗口

4. Canary 上线（flag ON for 5% 用户）
 └── 监控错误率、延迟、用户行为
 └── 对比指标：canary vs 基线
 └── 24-48 小时监控窗口
 └── 所有阈值通过才推进

5. 逐步增加（25% → 50% → 100%）
 └── 每步同样监控

6. 全量（flag ON for 所有用户）
 └── 监控 1 周
 └── 清理 feature flag

```


### 上线决策阈值

| 指标 | 推进（绿色） | 暂停调查（黄色） | 回滚（红色） |
|------|------------|---------------|-----------|
| 错误率 | 基线 10% 内 | 基线 10-100% 以上 | >2x 基线 |
| P95 延迟 | 基线 20% 内 | 基线 20-50% 以上 | >50% 基线 |
| 业务指标 | 中性或正向 | <5% 下降 | >5% 下降 |

### 什么时候回滚

立即回滚如果：

- 错误率增加超过 2x 基线
- P95 延迟增加超过 50%
- 用户报告问题突增
- 检测到数据完整性问题
- 发现安全漏洞

## 监控和可观察性

### 监控什么

```
应用指标：
├── 错误率（总数和按端点）
├── 响应时间（p50, p95, p99）
├── 请求量
├── 活跃用户
└── 关键业务指标

基础设施指标：
├── CPU 和内存使用
├── 数据库连接池使用
├── 磁盘空间
└── 网络延迟

```


### 上线后验证

上线后第一个小时：

```
1. 检查健康端点返回 200
2. 检查错误监控面板（无新错误类型）
3. 检查延迟面板（无回归）
4. 手动测试关键用户流程
5. 验证日志流动且可读
6. 确认回滚机制可用

```


## 回滚策略

每次部署前都需要回滚方案：

```markdown
## Rollback Plan for [Feature/Release]

### 触发条件
- 错误率 > 2x 基线
- P95 延迟 > [X]ms

### 回滚步骤
1. 禁用 feature flag（如果适用）
 或
1. 部署上一版本
2. 验证回滚
3. 通知团队

### 回滚时间
- Feature flag: < 1 分钟
- 重新部署上一版本: < 5 分钟
- 数据库回滚: < 15 分钟

```


## 常见合理化

| 合理化说法 | 现实 |
|---|---|
| "staging 能跑，生产也能跑" | 生产有不同的数据、流量模式和边界情况。部署后监控。 |
| "这个不需要 feature flag" | 每个功能都受益于 kill switch。 |
| "监控是开销" | 没有监控意味着你从用户投诉而不是面板发现问题。 |
| "回滚是承认失败" | 回滚是负责任的工程。发布坏功能才是失败。 |

## 危险信号

- 部署没有回滚方案
- 生产没有监控或错误报告
- 大爆炸式发布（一次全上，没有 staging）
- Feature flag 没有过期或归属
- 上线后第一个小时没人监控

## 验证

部署前：

- [ ] 上线前检查清单全部绿色
- [ ] Feature flag 已配置
- [ ] 回滚方案已文档化
- [ ] 监控面板已设置

部署后：

- [ ] 健康检查返回 200
- [ ] 错误率正常
- [ ] 延迟正常
- [ ] 关键用户流程可工作
- [ ] 日志在流动
