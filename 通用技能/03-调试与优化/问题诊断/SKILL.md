# Diagnose（诊断）

来源：Matt Pocock Skills 原版，英文入口保留在 `SKILL.original.md`。

这是处理疑难 Bug 的纪律。只有在理由明确时才跳过阶段。

探索代码库时，先用项目的领域词汇表建立相关模块的清晰心智模型，并检查你要触碰区域里的 ADR。

## 阶段 1：建立反馈循环

**这就是这个技能的核心。** 如果你有一个快速、确定、Agent 可运行的 Bug 通过/失败信号，你就能找到原因。二分、假设验证、插桩都只是在消费这个信号。没有这个信号，盯着代码看再久也救不了你。

在这里投入不成比例的精力。**要强硬。要有创造力。拒绝放弃。**

### 构造反馈循环的方法，按大致顺序尝试

1. **失败测试**：在能触达 Bug 的 seam 上写测试，可以是 unit、integration、e2e。
2. **Curl / HTTP 脚本**：对运行中的 dev server 发请求。
3. **CLI 调用**：用 fixture 输入运行命令，把 stdout 和已知正确快照做 diff。
4. **无头浏览器脚本**：用 Playwright / Puppeteer 驱动 UI，断言 DOM、console、network。
5. **回放捕获的 trace**：把真实网络请求、payload 或 event log 保存到磁盘，在隔离环境中穿过同一代码路径。
6. **一次性 harness**：启动系统的最小子集，用一个函数调用触发 Bug 路径。
7. **Property / fuzz 循环**：如果 Bug 是“有时输出错误”，跑 1000 个随机输入，寻找失败模式。
8. **二分 harness**：如果 Bug 出现在两个已知状态之间，自动化“启动状态 X、检查、重复”，让它能被 `git bisect run` 调用。
9. **差分循环**：让同一输入穿过旧版本和新版本，或两个配置，然后 diff 输出。
10. **HITL bash 脚本**：最后手段。如果必须有人点击，用 `scripts/hitl-loop.template.sh` 指挥人类，让循环仍然结构化。捕获输出后再反馈给你。

建立对的反馈循环，Bug 就已经修好了 90%。

### 迭代反馈循环本身

把反馈循环当成一个产品。只要有了一个循环，就继续问：

- 能不能更快？例如缓存 setup、跳过无关初始化、缩小测试范围。
- 信号能不能更尖锐？断言具体症状，而不是只断言“没有崩”。
- 能不能更确定？固定时间、固定 RNG seed、隔离文件系统、冻结网络。

一个 30 秒还 flaky 的循环，只比没有循环好一点。一个 2 秒且确定的循环，才是调试超能力。

### 非确定性 Bug

目标不是得到干净复现，而是提高复现率。把触发器循环 100 次、并行化、增加压力、缩小时序窗口、注入 sleep。50% 复现率的 flaky Bug 可以调；1% 不行。继续提高复现率，直到它能被调试。

### 如果你真的无法建立反馈循环

停下来，并明确说出来。列出你试过什么。向用户请求：

- 能复现的环境访问权限；
- 捕获物，例如 HAR 文件、日志 dump、core dump、带时间戳的录屏；
- 添加临时生产插桩的许可。

不要在没有反馈循环的情况下进入假设阶段。

在你拥有可信的反馈循环之前，不要进入阶段 2。

## 阶段 2：复现

运行反馈循环。看着 Bug 出现。

确认：

- [ ] 循环产生的是**用户描述的失败模式**，不是附近的另一个失败。调错 Bug 就会修错问题。
- [ ] 失败能在多次运行中复现；对于非确定性 Bug，复现率要高到足以支撑调试。
- [ ] 已捕获精确症状，例如错误消息、错误输出或慢在哪里，这样后面的阶段能验证修复确实命中问题。

复现之前不要继续。

## 阶段 3：提出假设

先生成 **3–5 个排序后的假设**，再测试任何一个。只生成一个假设会让你锚定第一个看起来合理的解释。

每个假设都必须**可证伪**：写清它做出的预测。

格式：如果 `<X>` 是原因，那么 `<改变 Y>` 会让 Bug 消失，或 `<改变 Z>` 会让 Bug 更严重。

如果不能说出预测，这个假设就只是感觉。丢掉，或继续 sharpen。

**测试前把排序后的列表展示给用户。** 用户经常有领域知识，可以立刻重新排序：“我们刚部署了和 #3 有关的改动。”他们也可能知道哪些假设已经被排除。这是低成本、高收益的检查点。不要被它卡住；如果用户 AFK，就按你的排序继续。

## 阶段 4：插桩

每个探针都必须对应阶段 3 中某个具体预测。**一次只改变一个变量。**

工具偏好：

1. 如果环境支持，优先用 **debugger / REPL 检查**。一个断点胜过十条日志。
2. 在能区分假设的边界加**定向日志**。
3. 永远不要“把所有东西都打日志然后 grep”。

**每条 debug log 都加唯一前缀**，例如 `[DEBUG-a4f2]`。最后清理时一次 grep 就能找到。没有标签的日志会活下来；带标签的日志应该死掉。

**性能分支。** 对性能回退来说，日志通常是错的。先建立基线测量，例如 timing harness、`performance.now()`、profiler、query plan，然后再二分。先测量，再修复。

## 阶段 5：修复 + 回归测试

先写回归测试，再写修复，但前提是存在**正确的 seam**。

正确的 seam 是：测试能以调用点真实发生的 Bug 模式触发问题。如果唯一可用的 seam 太浅，比如单调用方测试无法复现需要多个调用方的 Bug，或 unit test 无法复现触发链，那里的回归测试只会给你虚假的信心。

**如果没有正确的 seam，这本身就是发现。** 记录它。代码架构正在阻止你锁住这个 Bug。把它标记给下一阶段。

如果存在正确 seam：

1. 把最小复现转成该 seam 上的失败测试。
2. 看它失败。
3. 应用修复。
4. 看它通过。
5. 重新运行阶段 1 的反馈循环，验证原始的非最小场景。

## 阶段 6：清理 + 复盘

宣布完成前必须满足：

- [ ] 原始复现不再复现，重新运行阶段 1 的循环。
- [ ] 回归测试通过，或已记录缺少 seam 的事实。
- [ ] 所有 `[DEBUG-...]` 插桩都已删除，用 grep 检查前缀。
- [ ] 一次性原型已删除，或移动到明确标记的 debug 位置。
- [ ] 在 commit / PR message 中写明最终被证实的假设，让下一个调试的人能学习。

然后再问：什么能防止这个 Bug？如果答案涉及架构变化，例如没有好的测试 seam、调用方缠在一起、隐藏耦合，把具体信息交给 `/improve-codebase-architecture` 技能。这个建议要在修复之后给，不要在修复前给，因为现在你掌握的信息比一开始更多。
